Fun Facts

Things That'll Blow Your Mind

We know security sounds boring. It's not. Here's why.

🧲
The Only Way To Steal Your Key? Pull The RAM Chip.

Hackers typically steal passwords through software — keyloggers, malware, memory dumps. With GreenBox's VirtualLock, your encryption key lives exclusively in physical RAM, locked in place by the Windows kernel itself.

To steal it, an attacker would literally need to physically remove your RAM stick from your PC while it's running — then read it in milliseconds before it loses power. RAM is volatile — cut power and it forgets everything instantly. No data persists. Nothing survives. Your key vanishes like smoke.

Physics-Level Security
🏔️
100,000 Attempts Just To Check One Password

Imagine a hacker trying to guess your password. They try "password123". GreenBox forces them to do 100,000 iterations of PBKDF2 just to test that one guess. On a fast server doing 1 billion operations per second, they can only test ~10,000 passwords per second.

A standard dictionary has ~170,000 words. At that rate? They're done in 17 seconds. Add numbers + symbols? It's now centuries of compute time. Your pet's name + birth year is now Fort Knox.

100K Iterations
🌍
All Of Earth's Computers Can't Crack AES-256

There are roughly 10 billion computers on Earth. If every single one tested 1 trillion AES-256 keys per second (they can't, but humor us), and they started when the universe was born (13.8 billion years ago), they'd have checked 4.35 × 10³⁰ keys.

AES-256 has 1.16 × 10⁷⁷ keys. They've barely started. They'd need to repeat the age of the universe 10⁴⁶ more times. Your file is safe.

Mathematically Unbreakable
👻
RAM Is A Ghost — It Forgets Everything

RAM (Random Access Memory) is volatile — it needs constant electricity to remember anything. The moment power cuts, all data in RAM vanishes — including your encryption key.

This is why GreenBox uses VirtualLock: it keeps the key only in RAM and never lets Windows write it to the pagefile (swap file) on your hard drive — where it could persist and be recovered. No power = no key. It's gone. Forever.

Volatile by Design
🎲
Encrypted Files Look Like Pure Random Noise

AES-256 in CTR mode is so good that encrypted files are statistically indistinguishable from random data. Run every known test for patterns — you'll find none. There's no header saying "this is encrypted." No structure. No clues.

It's not just locked — it looks like the output of a nuclear explosion of randomness. Even knowing you used GreenBox gives an attacker exactly zero advantage.

Zero Patterns
🚫
One Tampered Bit = Total Refusal

What if someone flips a single bit in your encrypted file? Maybe tries to inject data? Corrupt your file to cause a crash?

GreenBox runs dual HMAC-SHA256 verification — checking the header, metadata, AND data before writing anything. One bit out of place? GreenBox detects it and refuses to output anything. No partial writes. No corruption. No exploits. Just a clean rejection.

Under The Hood
01
Password → PBKDF2 Key Derivation BCrypt + Windows Crypto
Your password is fed into PBKDF2 with 100,000 iterations, a cryptographically random 32-byte salt, and SHA-256 as the PRF. This transforms even a weak password into a full 256-bit key. The key is immediately VirtualLocked into RAM — pinned, unpageable, invisible to software.
02
Generate Random IV + Salt Windows CryptGenRandom
A fresh cryptographically secure random 128-bit Initialization Vector and 256-bit salt are generated for every single encryption operation. Even if you encrypt the same file twice with the same password, the outputs are completely different. This prevents pattern analysis.
03
Multi-threaded AES-256-CTR Encryption 1MB Chunks · Parallel
The file is split into 1MB chunks and distributed across CPU threads. Each chunk is encrypted with AES-256 in Counter Mode (CTR) — which turns AES into a stream cipher, allowing parallel processing and perfect forward secrecy per chunk. No padding oracle attacks possible.
04
Metadata Header Construction Timestamps Preserved
GreenBox builds a structured header containing: original file timestamps (real creation time + modification time), file size, metadata hash, and version info. When decrypted, your file is restored bit-for-bit including exact timestamps — files look untouched by time.
05
Dual HMAC-SHA256 Signing 2-Pass Verification
Pass 1: HMAC-SHA256 over the header + metadata. Pass 2: HMAC-SHA256 over the entire encrypted data block. Both signatures must verify before the output file is written. This catches tampering, corruption, wrong passwords, and partial write failures simultaneously.
06
Safe File Write via std::filesystem Bounds Checked · Atomic
Only after both HMAC checks pass does GreenBox write to disk using std::filesystem with full bounds checking and metadata validation. Keys are immediately zeroed from VirtualLocked RAM using SecureZeroMemory — leaving no trace of your key anywhere.
Cryptography

The Technical Fortress

Every algorithm chosen for a reason. Every parameter maximized for security.

🔐
AES-256-CTR
Primary Encryption Algorithm
  • 256-bit key — the largest AES key size, 14 rounds of transformation
  • Counter Mode (CTR) — converts block cipher to stream cipher, enables parallelism
  • No padding required — eliminates padding oracle attack surface entirely
  • Each 128-bit block gets a unique counter value — never repeats
  • NIST-approved, NSA Suite B compliant, used in classified government systems
Key Space: 2^256 = 1.16 × 10^77 possible keys
At 10^18 guesses/sec → ~3.7 × 10^51 years to crack
🏗️
PBKDF2
Password-Based Key Derivation Function 2
  • 100,000 iterations — OWASP recommends minimum 600,000 for SHA-256 (we'll be upping this)
  • 32-byte random salt — prevents rainbow table attacks completely
  • SHA-256 as pseudo-random function
  • Implemented via Windows BCrypt — battle-tested, kernel-validated
  • Output: 256-bit derived key — full entropy regardless of password strength
100,000 iterations × one guess = massive compute cost
Attacker speed: ~10,000 guesses/sec (vs millions unprotected)
✍️
HMAC-SHA256
Authentication + Integrity
  • 256-bit cryptographic Message Authentication Code
  • Pass 1: Signs header + metadata — detects structural tampering
  • Pass 2: Signs encrypted data — detects content tampering
  • Wrong key → wrong HMAC → immediate rejection, no output written
  • Provides authenticated encryption — not just confidentiality
Forge probability: 1/2^256 per attempt
Provides: Integrity + Authentication + Non-repudiation
🔒
VirtualLock + SecureZero
Memory Security Architecture
  • VirtualLock pins key pages in physical RAM — OS cannot page them to disk
  • SecureZeroMemory wipes key from RAM immediately after use
  • Windows Native Crypto API (BCrypt) — no third-party code in the crypto path
  • Prevents cold-boot attack via swap file — key never touches disk
  • Zero external DLL dependencies — no supply-chain attack surface
Key lifetime in RAM: milliseconds only
Key on disk: Never. Literally never. Not even 1 bit.

We've Heard Everything

Real questions from real people who were skeptical first.

Is it really free? What's the catch?
▼
There is no catch. GreenBox is completely free — no subscription, no trial, no "free tier with limits," no ads, no telemetry, no data collection. We built it because everyone deserves the right to privacy, not just people who can afford enterprise software. Use it personally. Use it commercially. Use it forever. The only thing we ask is nothing.
What happens if I forget my password?
▼
Your file is gone. And we mean that in the best possible way. GreenBox has absolutely no backdoor, no recovery mechanism, no "forgot password" option. Your password is the key. No key = no access. This is by design — a backdoor for you is a backdoor for attackers too. Write your password down. Use a password manager. Treat it like the key to a physical safe.
Can hackers break the encryption if they have the encrypted file?
▼
Mathematically speaking, no. AES-256 has never been broken and is not expected to be broken in our lifetimes or our grandchildren's lifetimes. The best known attack against AES-256 is still a brute force search, which would take longer than the age of the universe even with all computing power on Earth. Additionally, PBKDF2 with 100,000 iterations makes password guessing practically infeasible. The encrypted file is safe to store anywhere — cloud, USB, email, public server.
Does GreenBox send any data online? Does it phone home?
▼
Absolutely not. GreenBox makes zero network connections. Ever. There's no analytics, no telemetry, no update checker, no license verification, no crash reporting. Your files, your password, your machine. Nothing leaves your computer. You can run GreenBox on an air-gapped machine with no network connection and it works perfectly — because that's all it needs.
What file types can I encrypt?
▼
Any file. Every file. GreenBox treats all files as raw bytes — it doesn't care if it's a .docx, .jpg, .mp4, .zip, .exe, or a database file. The encryption process is identical for every file type. The only thing that matters is the bytes, and GreenBox handles them all the same — with the same military-grade encryption.
How fast is it? Will it slow down my PC?
▼
Very fast, and no. GreenBox uses multi-threaded processing with 1MB chunks distributed across your CPU cores. On a modern PC, you can expect several hundred MB/s for large files. The only "slow" part is PBKDF2 key derivation (intentionally, to resist attacks) — but that's a one-time ~1 second delay per file operation, not ongoing processing. Your system stays responsive throughout.
Is the RAM attack thing real? Should I worry about it?
▼
It's real but practically negligible for most users. The "cold boot attack" involves physically removing RAM and reading its residual charge before it dissipates. It requires the attacker to have physical access to your running PC, specialized forensic hardware, sub-second timing, and a clean room environment. For 99.9% of threat models — including corporate espionage, theft, hackers — you'll never face this. GreenBox already mitigates this further by keeping keys in RAM for milliseconds only via VirtualLock + SecureZeroMemory. Sleep soundly.
Why is it Windows only?
▼
GreenBox leverages Windows-native security primitives — VirtualLock, BCrypt (Windows CNG), SecureZeroMemory — which are Windows kernel APIs. These provide extremely deep, OS-level security guarantees. Building a cross-platform version would require different APIs on macOS and Linux, potentially weakening the security model. GreenBox was built right for Windows first. Maybe other platforms come later. Doing one thing perfectly beats doing many things adequately.
